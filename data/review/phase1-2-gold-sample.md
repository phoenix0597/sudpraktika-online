# Фаза 1.2 — эталонная выборка для ручной разметки

Статус: черновик выборки для юридической проверки.

Цель: выбрать 20–30 актов из первой партии, чтобы вручную проверить качество JSON v2,
донастроить `timeline`, `amounts`, `taxonomy.dispute_type` и получить few-shot примеры
для следующих партий.

## Что проверять вручную

Для каждого дела достаточно проверить не весь текст заново, а поля, которые будут влиять
на сайт и кластеризацию:

1. `taxonomy.dispute_type` — нормальный вид спора простым русским языком.
2. `taxonomy.claim_type` — тип требования без чрезмерно длинной фразы.
3. `claims_and_result.outcome.result_type` — привести к одному из нормальных значений:
   `satisfied`, `partially_satisfied`, `rejected`, `mixed`, `hold`.
4. `claims_and_result.amounts` — корректность заявленных/присуждённых сумм и основных статей.
5. `case_summary.timeline` — есть ли ключевые даты и не выдуманы ли они.
6. `case_summary.key_factors` — действительно ли эти факторы повлияли на исход.
7. `legal_analysis.legal_refs` — ссылки только из акта; без внешних номеров из памяти модели.
8. `publication.main_site_fit` / `publication.index_policy` — подходит ли дело для основного сайта.

## Минимальный формат ручной правки

Для каждого выбранного дела можно не переписывать весь JSON. Достаточно зафиксировать
правки в таком формате:

```md
## <docid>

- verdict: ok / needs_fix / exclude
- dispute_type_correct:
- claim_type_correct:
- result_type_correct: satisfied / partially_satisfied / rejected / mixed / hold
- amounts_fix:
- timeline_fix:
- key_factors_fix:
- legal_refs_fix:
- publication_fix:
- notes:
```

После ручной разметки эти правки нужно использовать для:

1. нормализации enum-значений в JSON;
2. few-shot примеров для `scripts/prompt_extract.txt`;
3. правил кластеризации Фазы 1.4.
