# Задача: первичная кластеризация 50 судебных актов по ситуациям

## Контекст

Проект: сайт по судебной практике о защите прав потребителей.

На текущем этапе подготовлены 50 структурированных JSON:

- `data/structured/structure_<docid>.json`

Enum-таксономия доведена до всех 50 JSON:

- во всех 50 есть `taxonomy.dispute_type_code`;
- во всех 50 есть `taxonomy.claim_type_codes`;
- `claim_type_codes` — массив строк;
- `publication.main_site_fit` — boolean;
- валидаторы проходят на 50/50.

Канонический словарь:

- `data/reference/zpp_enum_dictionary.json`

Отчёты по enum:

- `data/review/phase1-2-enum-suggestions.reviewed.csv`
- `data/review/phase1-4-enum-completion.csv`
- `data/review/phase1-4-enum-completion-by-antigravity.md`

## Цель

Сделать первичную кластеризацию 50 судебных актов по практическим ситуациям для будущей структуры сайта и pSEO-страниц.

Нужно получить не академическую классификацию, а рабочую карту:

- какие группы дел уже видны в выборке;
- какие группы могут стать посадочными страницами;
- какие дела нужно держать в `hold`;
- где кластер слишком тонкий и требует расширения выборки.

## Важный принцип

Сначала использовать простую и объяснимую группировку по уже готовым JSON-полям:

- `taxonomy.dispute_type_code`;
- `taxonomy.claim_type_codes`;
- `taxonomy.platform_or_company`;
- `taxonomy.object_type`;
- `taxonomy.situation_tags`;
- `claims_and_result.outcome.result_type`;
- `case_summary.key_factors`;
- `publication.index_policy`;
- `publication.main_site_fit`.

Embeddings на этом шаге не нужны, если без них получается понятная карта кластеров. Не усложнять без необходимости.

## Что нужно сделать

1. Прочитать все 50 файлов `data/structured/structure_*.json`.

2. Сформировать таблицу по каждому делу:

   - `docid`
   - `case_number`
   - `decision_date`
   - `region`
   - `source_url`
   - `dispute_type_code`
   - `claim_type_codes`
   - `platform_or_company`
   - `object_type`
   - `object_name`
   - `result_type`
   - `index_policy`
   - `main_site_fit`
   - `situation_tags`
   - 2–3 ключевых фактора из `case_summary.key_factors`

3. Сгруппировать дела в практические кластеры.

   Минимально ожидаемые группы:

   - недостатки товара / технически сложный товар;
   - дистанционная продажа и отказ от товара;
   - оплаченный товар не передали / задержка передачи;
   - недостоверная или неполная информация о товаре/продавце;
   - отказ от услуги и возврат денег;
   - работы/услуги с недостатками;
   - пограничные/нецелевые дела (`hold`).

   Можно предложить другие группы, если они лучше объясняют фактическую выборку.

4. Для каждого кластера определить:

   - `cluster_code` — короткий стабильный код на английском snake_case;
   - `cluster_name` — понятное русское название;
   - `case_count`;
   - список `docid`;
   - основные `dispute_type_code`;
   - основные `claim_type_codes`;
   - типичный пользовательский запрос/боль;
   - возможный угол будущей страницы;
   - приоритет:
     - `pillar` — крупная базовая страница;
     - `landing` — хорошая посадочная страница;
     - `long_tail` — узкий кластер, можно позже;
     - `hold` — не публиковать в основной индекс;
   - краткое обоснование.

5. Отдельно отметить дела с `publication.index_policy = hold` или `main_site_fit = false`.

6. Отдельно отметить спорные случаи, где кластер выбран неочевидно.

## Ограничения

- Не менять `user_story_*.md` и `practice_*.md`.
- Не менять `data/reference/zpp_enum_dictionary.json`.
- Не менять memory-файлы.
- Не делать git-коммит без отдельного указания пользователя.
- Не добавлять новые enum-коды.
- Не превращать кластеризацию в сложную ML-задачу без необходимости.

Если есть черновой скрипт `scripts/cluster_cases.py`, можно использовать или доработать его, но результат должен быть проверен по смыслу, а не просто принят автоматически.

## Критерии качества

Хороший результат:

- кластеры понятны человеку без юридического образования;
- кластеры пригодны для структуры сайта;
- названия не слишком юридизированы;
- не создаются микрокластеры из 1 дела без причины;
- маркетплейс/компания не становится главным кластером, если юридическая ситуация важнее;
- `hold`-дела не смешиваются с индексируемыми ЗоЗПП-страницами;
- выводы прямо опираются на JSON, а не на догадки.

## Проверки

После создания артефактов проверить:

- в CSV ровно 50 строк без учёта заголовка;
- каждый `docid` из `data/structured/structure_*.json` присутствует ровно один раз;
- дела с `publication.index_policy = hold` или `main_site_fit = false` имеют `page_priority = hold`;
- в Markdown-отчёте есть явные рекомендации по первым страницам.

Также повторно запустить:

```powershell
py scripts/validate_structures.py
py scripts/verify_all.py
```

Ожидаемый результат:

- `validate_structures.py`: 50/50 без ошибок;
- `verify_all.py`: 50/50, 0 подозрительных ссылок.
