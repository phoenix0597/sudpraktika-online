# Фаза 1.5: проектирование SSG-прототипа

## Решение

Для первого SSG-прототипа делаем 5 страниц-ситуаций и карточки дел внутри них. Отдельные страницы конкретных дел проектируем сразу в контракте, но реализацию можно отложить на второй шаг прототипа.

Цель прототипа — проверить полезность структуры и качество данных, а не сразу строить весь сайт.

## Выбранные страницы-ситуации

| Код кластера | Дел | Тип | Рабочий маршрут | H1 |
|---|---:|---|---|---|
| `goods_defect_art18` | 24 | pillar | `/praktika/nekachestvennyy-tovar/` | Некачественный товар: как вернуть деньги, взыскать неустойку и штраф |

| `service_refusal_art32` | 5 | landing | `/praktika/vozvrat-deneg-za-uslugu/` | Как отказаться от услуги и вернуть деньги |

Маршруты рабочие. Перед публичным запуском их можно адаптировать под финальный бренд и роутинг, но для прототипа они достаточно понятны и не привязаны к конкретному маркетплейсу.

## Шаблон страницы-ситуации

### 1. Hero

Поля:

- `cluster_name`
- рабочий H1 из таблицы выше
- количество дел в кластере
- короткое описание пользовательской проблемы
- отметка: «Практика собрана из судебных актов, ссылки на источники внутри карточек»

Назначение: быстро объяснить пользователю, что страница именно про его ситуацию.

### 2. Краткое резюме практики

Блоки:

- сколько дел в выборке;
- сколько дел удовлетворены полностью / частично / отказано / mixed;
- типичные требования (`claim_type_codes`);
- типичные объекты и компании.

Поля:

- `claims_and_result.outcome.result_type`
- `taxonomy.claim_type_codes`
- `taxonomy.platform_or_company`
- `taxonomy.object_type`

Примечание: не формулировать как статистически репрезентативную судебную аналитику. Это «по нашей выборке актов».

### 3. Что обычно требует потребитель

Показать chips/список требований:

- возврат цены;
- неустойка;
- моральный вред;
- штраф 50%;
- убытки;
- расходы;
- передача товара;
- возврат страховой премии и т.п.

Поля:

- `taxonomy.claim_type_codes`
- `claims_and_result.remedy`
- `claims_and_result.amounts.items_claimed`

### 4. Сколько взыскивают

Мини-сводка по суммам:

- диапазон `awarded_total`;
- медиана/типичные значения — можно считать позже;
- отдельные строки по `items_awarded`, если они заполнены.

Поля:

- `claims_and_result.amounts.awarded_total`
- `claims_and_result.amounts.items_awarded`
- fallback: если `claimed_total` пустой, не показывать общий размер требований, а показывать детализацию из `items_claimed`.

Важно: `claimed_total` заполнен не во всех делах, поэтому шаблон не должен зависеть от него.

### 5. Почему суд удовлетворяет или отказывает

Показать 3–6 повторяющихся факторов по делам кластера.

Поля:

- `case_summary.key_factors`
- `legal_analysis.arguments_rejected`
- `legal_analysis.holding`

Формат: простые тезисы, без длинных юридических цитат.

### 6. Карточки дел

Одна карточка = одно дело. Минимальный состав:

- бытовое название/summary;
- суд, регион, дата, номер дела;
- объект и компания;
- результат;
- присуждённая сумма;
- 1–2 ключевых фактора;
- ссылка на первоисточник;
- ссылка на будущую страницу дела.

Поля:

- `source.docid`
- `source.source_url`
- `court.case_number`
- `court.court_name`
- `court.region`
- `court.decision_date`
- `taxonomy.platform_or_company`
- `taxonomy.object_type`
- `taxonomy.object_name`
- `case_summary.situation`
- `case_summary.key_factors`
- `claims_and_result.outcome.result_type`
- `claims_and_result.amounts.awarded_total`

Рекомендуемая сортировка в прототипе:

1. индексируемые дела (`publication.index_policy = index`);
2. более свежие решения;
3. дела с непустыми `awarded_total`;
4. дела с более понятными `object_type`/`platform_or_company`.

### 7. Фильтры внутри страницы

Фильтры делать клиентскими, без генерации отдельных индексируемых URL.

Минимальный набор:

- результат дела;
- компания/платформа;
- тип объекта;
- требования;
- регион.

Не делать отдельные SEO-страницы по фильтрам на первом шаге. Гео/судебные страницы — только после накопления достаточного числа дел.

### 8. FAQ / чек-лист

Для прототипа лучше чек-лист, а не большой FAQ:

- подходит ли ситуация под этот тип спора;
- что проверить в документах;
- какие требования обычно заявляют;
- когда есть риск отказа.

FAQPage schema можно добавить позже, когда вопросы будут выверены редакционно.

### 9. Связанные ситуации

Ссылки между 5 страницами:

- некачественный товар ↔ дистанционный возврат;
- дистанционный возврат ↔ недостоверная информация;
- оплаченный товар не передали ↔ дистанционный возврат;
- отказ от услуги ↔ недостоверная информация, если услуга навязана/описана неверно.

### 10. Disclosure и доверие

На странице показать:

- «Материал подготовлен на основе судебных актов»
- «Обработка данных выполнена алгоритмически; ссылки на первоисточники сохранены»
- статус проверки: `publication.human_review_status`
- ссылка на методологию проекта, когда она появится.

## Шаблон страницы дела

Реализация может быть вторым шагом, но контракт сразу фиксируем.

Маршрут:

- `/dela/<docid>/`

Блоки:

1. H1: бытовое название дела.
2. Карточка источника: суд, дата, номер дела, регион, ссылка на акт.
3. Ситуация простым языком.
4. Что требовал потребитель.
5. Что решил суд и какие суммы взыскал.
6. Почему суд так решил.
7. Ошибки/факторы, повлиявшие на исход.
8. Хронология, если `timeline` заполнен.
9. Ссылка обратно на страницу-ситуацию.

Проверено по 46 делам выбранных 5 кластеров:

- `timeline` заполнен у 45/46 дел;
- `amounts` есть у 46/46 дел;
- `items_claimed` есть у 46/46 дел;
- `items_awarded` есть у 43/46 дел;
- `awarded_total` есть у 44/46 дел;
- `key_factors` есть у 46/46 дел;
- `practical_takeaways` есть у 46/46 дел;
- `legal_analysis` есть у 46/46 дел.

Вывод: данных достаточно для SSG-прототипа.

Ограничения:

- `claimed_total` часто пустой, поэтому не делать его обязательным;
- у дела `TZsI5FbnZOZJ` нет timeline, нужен fallback «хронология не выделена»;
- у части дел `platform_or_company` пустое или «не указан», фильтр должен это корректно обрабатывать;
- `amounts` использовать осторожно: показывать как извлечённые из акта, без обещания калькуляторной точности.

## Минимальный data contract для генератора

Генератор страниц должен читать:

- `data/review/phase1-4-case-clusters.csv` — карта кластеров и page_priority;
- `data/structured/structure_<docid>.json` — основное содержание карточек и страниц;
- `data/review/phase1-4-cluster-validation.md` — редакционное решение по первым страницам.

Для каждой страницы-ситуации генератор собирает дела по `cluster_code` и исключает:

- `page_priority = hold`;
- `publication.index_policy = hold`;
- `publication.main_site_fit = false`.

## Минимальная структура компонентов

Если стек будет Astro/Next, достаточно следующих компонентов:

- `SituationPage`
- `PracticeSummary`
- `ClaimChips`
- `AmountsSummary`
- `CaseCard`
- `CaseFilters`
- `RelatedSituations`
- `SourceDisclosure`

Для прототипа можно начать без интерактивного калькулятора. Калькулятор неустойки — отдельный блок после проверки шаблона и данных.

## Следующий практический шаг

Сделать генератор/прототип, который из текущих JSON и CSV создаёт 5 Markdown/HTML-страниц-ситуаций. Перед этим полезно решить стек прототипа:

1. самый быстрый путь — Markdown/HTML-генерация Python-скриптом;
2. более близко к будущему сайту — Astro;
3. Next.js оставить на этап полноценного фронтенда, если появится необходимость в сложной интерактивности.

Рекомендация: для проверки структуры начать с Python → Markdown/HTML. Это быстрее, дешевле по времени и не блокирует будущий переход на Astro/Next.
