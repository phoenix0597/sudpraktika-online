# Техническая Спецификация MVP — zpp-consult

Канонический источник требований к продукту и реализации.
Долговременные архитектурные правила и рабочие соглашения — в `assistant-memory.md`.
Обоснование решений, риски и исследовательская база — в `memory/topic-product-strategy.md`.

## Глобальная Цель Проекта

Создать масштабируемый каталог судебной практики по защите прав потребителей простым бытовым языком, генерирующий стабильный органический трафик и конвертирующий его в доход, в первую очередь — через размещение контекстной рекламы (РСЯ) на сайте, а также через дополнительные каналы монетизации — без публичного лица и без самостоятельного оказания юридических услуг.

## Суть Продукта

Каталог судебной практики по защите прав потребителей (ЗоПП), изложенный **простым бытовым языком**, на основе массового парсинга судебных актов. Каждый акт обрабатывается: из него выделяется жизненная ситуация, способ защиты, вклад в судебную практику и уникализирующие блоки ценности. Каталог публикуется как pSEO-сайт с поэтапным масштабированием.

Это НЕ ИИ-сервис юридических консультаций (исходная гипотеза отозвана на этапе планирования). Это медиа/каталог с монетизацией трафика.

## Принятые Стратегические Решения

| Развилка | Решение |
| --- | --- |
| E-E-A-T / «Корней» | Профиль безликой «юридической редакции» + методология + отметки проверки + верифицируемость по номерам дел |
| Монетизация (старт) | **РСЯ/Я.Директ** как основной канал + лиды юристам + партнёрки; DOCX-генератор — резервный/пост-MVP канал |
| Масштаб запуска | Поэтапный: пилот 100–300 страниц → масштабирование +500–1000/мес после валидации реакции поисковиков |
| Источник данных | Официальные страницы судов в ГАС «Правосудие» (sudrf.ru). Программный доступ к **отдельным судебным актам** (не к базе целиком) по их реквизитам/ссылкам. Единичный акт как официальный документ не охраняется АП (ст. 1225 ГК РФ); право изготовителя базы (ст. 1334 ГК РФ) защищает компиляцию, а не единичный документ, поэтому не затрагивается. Суды публикуют акты уже обезличенными в силу Закона №262-ФЗ — повторное обезличивание не требуется. Юридическую оценку даёт пользователь-юрист |
| Пилотный сегмент | **Маркетплейсы (Озон/ВБ) + процессуальные ситуации** (неустойка, претензия, иск) — решение 2026-06-24 по данным рыночного замера Wordstat API (спрос 112K + 107K показов/мес). Ранее (2026-06-23) пилотом были технически сложные товары (спрос 16K) — опровергнуто замером. Порядок масштабирования остальных вертикалей — в `memory/topic-product-strategy.md` (Фаза 7) |
| Доменная стратегия | Основной домен без субдоменов; индексация с нуля для Яндекса. Опц. подклейка дроп-домена для Google в будущем |
| Публичное лицо | Отсутствует. Пользователь — юрист, но не выступает лицом проекта |

## Иерархия Контента (Модель Данных)

Три уровня:

1. **Категория** — верхний уровень (товары / услуги / финансовые / жильё / ЖКХ / туризм).
2. **Ситуация** — конкретная бытовая фабула; **главная целевая страница** под кластер ключевых слов. Один кластер актов → одна страница-ситуация.
3. **Дело** — конкретный судебный акт; страница под long-tail, ссылается на свою «ситуацию».

Два типа страниц с разным интентом: страница-ситуация собирает коммерческие/информационные кластеры запросов; страница-дело ловит длинный хвост и накачивает страницу-ситуацию внутренней перелинковкой.

## Схема Извлечения Из Акта (JSON)

Для каждого акта создаётся `data/structured/structure_<docid>.json` по схеме v2.
Цель JSON — не заменить текстовые материалы, а дать стабильные поля для кластеризации,
фильтров, карточек дел, генерации страниц и контроля индексации.

Обязательные блоки JSON v2:

1. **`source`** — технический якорь первоисточника: `docid`, `source_url`, домен,
   путь к сырому акту, хэш текста. Эти поля заполняются детерминированно на этапе
   парсинга/очереди, а не «вспоминаются» LLM.
2. **`court`** — номер дела, суд, уровень суда, судебная система, регион, дата,
   инстанция. Регион и суд нужны для фильтров и возможных служебных страниц, но
   такие страницы индексируются только при достаточном числе дел и уникальном
   полезном содержании.
3. **`taxonomy`** — минимальная таксономия: правовая область, тип процесса,
   аудитория, вертикаль, нормализованный **код вида спора** (`dispute_type_code`),
   человекочитаемый вид спора (`dispute_type`), нормализованные коды требований
   (`claim_type_codes`), человекочитаемый тип требования (`claim_type`),
   платформа/компания, объект спора, теги ситуации. Словари значений закреплены
   в `data/reference/zpp_enum_dictionary.json`: `dispute_type_code` строится
   преимущественно от правовой конструкции ЗоЗПП, а не от маркетплейса или товара.
   Ситуационные детали остаются в `platform_or_company`, `object_type`,
   `object_name` и `situation_tags`. Это даёт масштабирование без тяжёлой онтологии.
4. **`parties`** — фокусная сторона (для текущего MVP обычно потребитель),
   оппоненты и третьи лица. Поле сделано нейтральным, чтобы позже не ломать схему
   при добавлении других правовых вертикалей.
5. **`case_summary`** — жизненная ситуация, хронология, ключевые факторы,
   практические выводы и необычные моменты.
6. **`claims_and_result`** — способ защиты, результат для фокусной стороны,
   заявленные и присуждённые суммы.
7. **`legal_analysis`** — holding, правовые ссылки и отвергнутые доводы.
   **Анти-галлюцинация (критическое правило, закреплено 2026-06-25):**
   LLM извлекает только те правовые ссылки (статьи, дела, постановления, пункты),
   которые буквально присутствуют в тексте акта. Запрещено генерировать ссылки
   на внешнюю практику, дела, номера или пункты постановлений из «знаний» модели.
   После извлечения анализ прогоняется через верификатор (`scripts/verify_citations.py`);
   ссылки, не найденные в тексте акта, флагуются и отбраковываются.
8. **`publication`** — простое решение для поисковых рисков: подходит ли дело
   основному сайту, можно ли индексировать страницу, нужна ли ручная проверка.
   SEO-оценки, «баллы качества» и прогнозы ранжирования в JSON не хранятся.
9. **`processing`** — технический статус обработки JSON (`metadata_seeded`,
   `migrated_from_v1_needs_review`, `complete`) и модель/дата обработки.

Арбитражные/хозяйственные дела на старте не смешиваются с гражданским ЗоПП-каталогом
для обычных людей. Если такие дела попадают в сбор, они помечаются
`publication.main_site_fit: false` и `publication.index_policy: "hold"` до отдельного
решения о новой вертикали.

### Служебные SEO-страницы (гео/суд long-tail)
Группировка по регионам и судам — потенциальные отдельные страницы с собственным
спросом («судебная практика по возврату Озон в Москве», «решения Никулинского суда
по ЗоПП»). Они создаются из того же датасета, что и основные страницы-ситуации,
но индексируются только при достаточном числе дел и уникальной сводке, а не как
массовые пустые фильтры. Маршруты-кандидаты: `/region/<регион>/<ситуация>/`,
`/court/<суд>/<ситуация>/`. Реализуется в Фазе 1.5 (генерация страниц).

Регион и конкретный суд также являются продуктовой ценностью: пользователь и юрист
должны видеть, как похожие ситуации разрешаются в разных регионах и судах. На старте
это фильтры и агрегаты внутри страниц-ситуаций; отдельные индексируемые страницы
создаются позже, только когда есть достаточная выборка и самостоятельная полезная
сводка.

### Инстанционный слой практики

Первостепенная глобальная задача — покрытие судебной практики судов общей юрисдикции:
первая инстанция, апелляция, кассация, Верховный Суд РФ и, когда применимо к вопросу,
Конституционный Суд РФ. Практика вышестоящих инстанций должна связываться с той же
категорией и ситуациями, что и низовая практика: пользователь видит не только решения
районных/городских судов, но и то, как подход корректируется или подтверждается выше.

Арбитражные суды — второй глобальный этап. Их не смешиваем с гражданским ЗоПП-разделом
для обычных пользователей без отдельной вертикали. ВАС РФ учитывается при необходимости
для будущих арбитражных/хозяйственных категорий и исторической практики, но не является
первичным источником для текущего ЗоПП-MVP.

## Структура Страниц

### Страница-ситуация (целевая)

1. H1 = бытовая формулировка ситуации
2. Резюме практики по кластеру (агрегат: % выигрышей, типичные суммы, частые доводы)
3. Свод рекомендаций по текущей выборке: что обычно важно для суда, какие доказательства помогают, какие ошибки повторяются, какие требования чаще удовлетворяются/отклоняются; при достаточном числе дел — региональные и судебные различия. Блок формулируется как обобщение выборки, не как индивидуальная юридическая консультация.
4. Ссылки на конкретные дела кластера (аннотированные карточки)
5. Интерактивный блок: мини-калькулятор неустойки/штрафа 50% или чек-лист «годится ли моя ситуация»
6. CTA: «Нужен юрист по аналогичному делу?» (монетизация — лид-форма) / вопрос редакции
7. FAQ-блок (schema FAQPage) из реальных вопросов по кластеру
8. Ссылки на родственные ситуации

### Страница-дела (long-tail)

1. H1 = бытовое название дела
2. Карточка верификации: суд, дата, номер дела, инстанция, ссылка на первоисточник
3. Жизненная ситуация бытовым языком
4. Что решил суд + суммы
5. Holding (вклад в практику)
6. Блок «Что сделано не так и как поступить правильно» (простым языком)
7. Ссылка наверх на «ситуацию» + родственные дела

## E-E-A-T Каркас (через Профиль Редакции)

Доверение строится на верифицируемости и прозрачности процесса, а не на авторе-человеке:

1. **Страница «О редакции»** — коллективный профиль «юридическая редакция», без обязательных имён, с описанием квалификации и принципа работы.
2. **Страница методологии** — как отбираются дела, переформулируются, проставляются отметки проверки, периодичность обновления.
3. **Отметки на каждой странице — честная перстраничная маркировка:** страница помечается либо «Обработано алгоритмом» (без ручной проверки), либо «Дополнительно проверено юристом — [дата]». Единая blanket-плашка «верифицировано юристом» на выборочно проверенных страницах недопустима (операционно несовместима с темпом публикации). Честное disclosure (ИИ участвует в обработке) усиливает доверие и снижает риск пессимизации за скрытый ИИ-контент.
4. **Якорь первоисточника** — кликабельный номер дела/ссылка на суд; каждый тезис верифицируем.
5. **Контакт и редакционная почта** для исправлений — сигнал «живого» проекта.

ЗоПП — менее YMYL-критичная ниша, чем уголовное/семейное право, поэтому требования к автору мягче.

## Премиальный UI как фактор доверия и поведенческих метрик

Дизайн напрямую влияет на доверие в YMYL-нише и на поведенческие сигналы «Корнея» (время на странице, отказы, глубина скролла): чистая типографика, строгая цветовая палитра, аккуратные карточки дел, отсутствие визуального шума. Премиальный вид снижает bounce-rate и повышает time-on-page — прямые факторы ранжирования Яндекса.

## Стек (Предварительно)

- **Фронтенд/рендеринг:** статическая генерация (SSG) для мгновенной загрузки страниц (важно для LCP и поведенческих факторов). Варианты: **Next.js (App Router)** или Astro; WordPress — только если SSG-плагин или headless-подход. Решение зафиксировать на Фазе 3.
- **Хранилище данных:** JSON/SQLite/Postgres для актов и кластеров.
- **Генерация страниц:** JSON → статические HTML-страницы по шаблонам (отдельный шаблон для «ситуации» и «дела»).
- **Генерация DOCX:** пост-MVP; серверный модуль (python-docx / docx-templater), заполняющий шаблоны претензии/иска плейсхолдерами из данных дела и пользовательского ввода.
- **Платежи:** пост-MVP (нужны только для DOCX-генератора; каналам MVP — РСЯ, лиды, партнёрки — платёжный шлюз не требуется).
- **Извлечение структуры (mailbox-путь, закреплено 2026-06-25; уточнено 2026-06-27):** основной масштабируемый путь для больших партий — внешний ИИ-агент с собственной моделью (Antigravity, Mimo, ZCode, Codex или другой). Интеграция через «почтовый ящик» (контракты на обмен файлами, без жёсткой связки): Python-пайплайн наполняет `data/inbox/_queue.json` (статус pending) и технические source-метаданные, ИИ-агент читает `data/inbox/_TASK.md` (единая инструкция: роль, задача, анти-галлюцинация, формат), обрабатывает акты, пишет `user_story_<docid>.md` + `practice_<docid>.md` + `structure_<docid>.json` в `data/structured/`, проставляет status: done только после полного комплекта. Внешний LLM API (например, DeepSeek API) и скрипты, которые его вызывают, используются только при явном разрешении пользователя в конкретной задаче; автоматическим fallback для агента они не являются. Вывод любого агента проходит через единые проверки `verify_citations.py` и `validate_structures.py`.
- **Технический SEO:** canonical URL, sitemap.xml, robots.txt, Schema.org (WebSite/Organization/Article, FAQPage, BreadcrumbList), быстрый LCP, внутренняя перелинковка, человекопонятные URL-слаги. Canonical/sitemap/robots генерируются из публичного `SITE_PUBLIC_URL`. Внутренние ссылки могут оставаться относительными на этапе прототипа; перед production отдельно решить, оставлять их такими или перейти на root-relative маршруты вида `/praktika/...`. HowTo rich results сняты Google; FAQ rich results ограничены авторитетными gov/мед-сайтами — на rich-results не рассчитываем, FAQPage оставляем для структуры/LLM. `LegalService` некорректен для сайта, который не оказывает юридические услуги.

## Монетизация

**Приоритетный и основной канал — размещение контекстной рекламы (РСЯ) на сайте.** Это центральный фокус проекта (решение 2026-06-24): строим генератор трафика → монетизируем через РСЯ → масштабируем расширением правовой тематики. Доход издателя начисляется за видимые показы по ставке Яндекса, а не из CPC рекламодателя (CPC ≈240–680 ₽ — это ставка аукциона, не выплата партнёру). Экономику моделируем через фактический RPM с тремя сценариями (пессимистичный/базовый/оптимистичный). Цель — стремиться к оптимистичному прогнозу: при ~1 200 уник./день и RPM ~300 ₽ доход ~54 000 ₽/мес (детально — `memory/topic-market-sizing.md`). РСЯ подключается после появления измеримого трафика.

**Каналы на будущее (не активируются на старте, по решению 2026-06-24):**
- **Лиды юристам** — отложен; активируется позже, когда трафик и данные будут стабильны.
- **Партнёрки** (СберПраво/ЕЮС и т.п.) — отчисления за направление.
- **DOCX-генератор пакета документов** (претензия + иск + инструкция) — пост-MVP. Разовая продажа 300–1500 ₽. Вынесен за рамки MVP для снижения сложности реализации.

**Стратегия масштабирования:** достигнутый успех на ЗоПП (генератор трафика + РСЯ) реплицируется добавлением новых правовых вертикалей — дела по другим темам права обрабатываются тем же пайплайном, расширяя охват спроса. Порядок тем — по рыночному замеру спроса (`memory/topic-demand-raw-data.md`).

## Воронка Конверсии (логика CTA)

1. Пользователь ищет решение проблемы в поисковике (long-tail запрос) → попадает на SEO-страницу.
2. На странице-ситуации видит релевантную позицию суда по аналогичному делу: фабула → закон → итог.
3. CTA: «Нужен юрист по аналогичному делу?» — лид-форма для передачи контакта практикующему специалисту.
4. Пользователь оставляет заявку → лид передаётся юристу-партнёру → комиссия проекту.
5. (Пост-MVP) Альтернативный CTA: «Сгенерировать пакет документов (претензия + иск + инструкция) по аналогичному случаю» → оплата → скачивание DOCX-пакета.
