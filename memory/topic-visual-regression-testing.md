# Визуальное и регрессионное тестирование SSG-страниц

## Задача

При наполнении сайта и изменении структуры страниц нужно автоматически проверять, что:

- новые страницы корректно генерируются;
- старые страницы не ломаются после изменений шаблонов/CSS;
- визуальные дефекты ловятся без ручного просмотра пользователем;
- дизайн остаётся единообразным.

Ограничение: решение должно быть простым, дешёвым и не создавать лишнюю инфраструктуру.

## Мини-ресерч

Рассмотренные варианты:

1. **Playwright visual comparisons**  
   Playwright Test умеет создавать эталонные скриншоты и сравнивать последующие прогоны с ними через `toHaveScreenshot()`. Это бесплатный встроенный механизм без внешнего сервиса. Документация: https://playwright.dev/docs/test-snapshots

2. **Playwright в CI / GitHub Actions**  
   Playwright официально поддерживает запуск в CI, включая GitHub Actions, HTML-отчёты и трассировки. Документация: https://playwright.dev/docs/ci-intro и https://playwright.dev/docs/ci

3. **BackstopJS**  
   Специализированный open-source инструмент визуального регрессионного тестирования. Поддерживает Chrome Headless и сценарии с Playwright/Puppeteer. Документация: https://garris.github.io/BackstopJS/  
   Минус для проекта: отдельная конфигурация и ещё один инструмент поверх Playwright.

4. **Percy / BrowserStack Percy, Chromatic, Applitools и аналоги**  
   Удобны для командного review визуальных изменений и кросс-браузерных снимков, но требуют внешнего сервиса, токенов и потенциальных расходов. Для текущего MVP преждевременно. Percy docs: https://www.browserstack.com/docs/percy/overview/visual-testing-basics

## Решение для проекта

Стартовать с **Playwright Test + встроенные screenshot snapshots**.

Причины:

- бесплатно;
- минимум новой инфраструктуры;
- подходит для локального запуска и GitHub Actions;
- закрывает и smoke/e2e-проверки, и визуальные регрессии одним инструментом;
- не требует отправлять страницы/скриншоты во внешний сервис;
- хорошо сочетается с будущим локальным веб-сервером.

BackstopJS и Percy не внедрять на старте. Вернуться к ним только если Playwright-снапшотов станет недостаточно: много ложных срабатываний, нужен удобный веб-review, кросс-браузерная матрица или командная история визуальных изменений.

## Минимальная архитектура тестов

### 1. Локальный сервер

SSG-прототип нужно проверять через `http://localhost/...`, а не `file://`.

На старте достаточно простого статического сервера:

```powershell
python -m http.server 4173 -d site_prototype
```

Позже это можно заменить на `npm`-скрипт, если появится полноценный фронтенд-стек.

### 2. Smoke-тесты структуры

Проверяют не пиксели, а базовую работоспособность:

- главная страница открывается;
- все ссылки на страницы-ситуации и страницы дел ведут на существующие HTML;
- на странице есть ровно один `h1`;
- нет служебных строк типа `refund`, `penalty_fine`, `goods_defect_art18` в пользовательском интерфейсе;
- нет слова `Прототип` в публичной части;
- есть дисклеймер;
- на страницах-ситуациях есть фильтры, карточки дел и блок рекомендаций по выборке;
- на страницах дел есть ссылка на первоисточник, итог дела, история и правовой разбор.

Эти тесты должны запускаться часто и быть стабильными.

### 3. Визуальные snapshot-тесты

Проверяют ограниченный набор эталонных страниц:

- главная страница;
- все 5 текущих страниц-ситуаций;
- 5–8 репрезентативных страниц дел:
  - длинный H1;
  - есть/нет timeline;
  - много норм в правовом разборе;
  - много требований/сумм;
  - типичный товарный спор;
  - услуга/работы;
  - некачественная информация.

Не нужно делать скриншот каждого дела. При тысячах страниц это будет медленно и шумно. Полное покрытие обеспечивается smoke-тестами и валидаторами данных, а визуальные снапшоты ловят поломку шаблонов.

### 4. Проверка адаптива

Для визуальных снапшотов достаточно двух viewport:

- desktop: 1366×900;
- mobile: 390×844.

Tablet можно добавить позже, если появятся реальные проблемы.

### 5. Правило обновления эталонов

Если дизайн меняется намеренно:

1. локально прогнать тесты;
2. посмотреть HTML-report/diff;
3. если изменения ожидаемые — обновить snapshots;
4. закоммитить snapshots вместе с изменением шаблона/CSS.

Если diff неожиданный — чинить шаблон/CSS.

## Улучшение работы с дизайном

Чтобы меньше ломать стиль:

1. Не править сгенерированные HTML вручную. Источник правды — `scripts/generate_ssg_prototype.py`.
2. Держать общие визуальные параметры в одном месте: CSS-переменные `:root`, базовые классы `panel`, `grid`, `case-card`, `summary-list`, `chips`.
3. Новые блоки сначала собирать из существующих классов. Новый CSS добавлять только если существующего набора явно недостаточно.
4. Для повторяющихся паттернов завести в генераторе маленькие render-функции:
   - `render_panel(...)`;
   - `render_summary_list(...)`;
   - `render_disclosure(...)`;
   - `render_case_facts(...)`.
5. На этапе перехода к финальному фронтенду вынести дизайн-токены отдельно: цвета, отступы, радиусы, типографика.

## Очередь внедрения

### Шаг 1. Подготовка

- Настроить локальный сервер для `site_prototype/`.
- Добавить минимальный `package.json`.
- Установить `@playwright/test`.
- Создать `playwright.config.ts`.

### Шаг 2. Smoke-тесты

Создать тесты:

- `tests/smoke/site.spec.ts`;
- проверка маршрутов, H1, обязательных блоков, пользовательских текстов.

### Шаг 3. Визуальные тесты

Создать:

- `tests/visual/pages.spec.ts`;
- список эталонных страниц;
- desktop/mobile snapshots.

### Шаг 4. CI

После решения по GitHub:

- добавить GitHub Actions;
- запускать генерацию прототипа, валидаторы данных, smoke-тесты и visual-тесты;
- сохранять HTML-report как artifact.

### Шаг 5. Расширение

Добавлять внешние сервисы типа Percy/Chromatic только при реальной необходимости.

## Принцип бритвы Оккама

Для MVP достаточно:

```text
Python генерация сайта
→ локальный статический сервер
→ Playwright smoke tests
→ Playwright visual snapshots
→ GitHub Actions artifact с отчётом
```

Не добавлять:

- Storybook;
- BackstopJS;
- Percy/Chromatic;
- кросс-браузерную матрицу;
- скриншоты каждой страницы;
- отдельную дизайн-систему как самостоятельный проект.

Эти инструменты могут понадобиться позже, но сейчас они увеличат сложность быстрее, чем пользу.
