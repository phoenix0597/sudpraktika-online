# План наполнения веб-сервиса судебной практикой

## Назначение

Этот файл фиксирует живой план наполнения сайта судебными актами: сколько дел нужно, какие ситуации усиливать, когда создавать новые страницы, как учитывать регионы, суды и инстанции.

Это не жёсткое ТЗ. План можно корректировать по мере появления данных, реакции поисковиков и изменения продуктовой стратегии.

## Базовый принцип

Наполнение строится не от абстрактного числа страниц, а от полезности срезов:

1. Сначала набираем достаточную практику по жизненным/правовым ситуациям.
2. Затем усиливаем ситуации регионами, судами и инстанциями.
3. Только после этого масштабируем число страниц.

Главный риск — создать много тонких страниц без самостоятельной пользы. Поэтому новые страницы появляются только при достаточной выборке и наличии уникальной сводки.

## Главная единица наполнения — ситуация

Для текущего ЗоПП-раздела не нужно равномерно покрывать всё право потребителей. Приоритет получают ситуации, где есть:

- поисковый спрос;
- понятная жизненная фабула;
- повторяемость судебной практики;
- возможность сделать полезную сводку;
- потенциал для фильтров и аналитики по требованиям, исходам, регионам и судам.

Рабочие пороги по количеству публикационно пригодных дел:

| Уровень | Количество дел | Решение |
|---|---:|---|
| 1–2 | слишком мало | копим, отдельную страницу не создаём |
| 3–5 | кандидат | держим в резерве, добираем |
| 6–10 | минимальная страница | можно публиковать осторожно |
| 10–25 | нормальная страница | есть основа для сводки, фильтров и рекомендаций |
| 25+ | сильный кластер | добирать точечно: регионы, суды, подтипы, инстанции |

Эти пороги не абсолютные. Если тема имеет высокий спрос или высокую продуктовую ценность, её можно добирать приоритетно. Если выдача даёт мусор, лучше остановиться, чем расширять слабую выборку.

## Кандидаты на новые ситуации ЗоЗПП

Текущий словарь `data/reference/zpp_enum_dictionary.json` остаётся рабочим словарём пилота. По итогам сверки с конструкциями ЗоЗПП 2026-06-27 новые коды не добавляются автоматически: сначала нужно увидеть повторяемые дела в корпусе.

При очередных партиях парсинга агенту полезно специально отслеживать следующие кандидаты:

| Кандидат | Правовая основа | Когда выделять отдельно |
|---|---|---|
| `proper_quality_goods_exchange_art25` | ЗоЗПП ст. 25 | Обмен/возврат товара надлежащего качества в обычной офлайн-продаже; не смешивать с дистанционной продажей по ст. 26.1. |
| `unfair_terms_imposed_services_art16` | ЗоЗПП ст. 16 | Недопустимые условия договора, навязанные услуги, платные допуслуги без согласия, ограничения прав потребителя; не смешивать с обычным отказом от услуги по ст. 32. |
| `harm_from_defect_art14` | ЗоЗПП ст. 14 | Вред жизни, здоровью или имуществу из-за недостатка товара, работы или услуги; потенциально чувствительная категория, требует аккуратной проверки. |
| `consumer_material_damage_art35` | ЗоЗПП ст. 35 | Утрата или повреждение вещи/материала потребителя при выполнении работы: ремонт, химчистка, изготовление из материала потребителя и т.п. |

Правило для агентов: если такие дела встречаются единично, фиксировать их в отчёте партии как кандидатов и не менять enum-словарь. Если находится повторяемый массив, предложить обновление `data/reference/zpp_enum_dictionary.json` отдельным шагом и вынести это в `batch_state.md` как требующее решения пользователя. До обновления словаря не записывать кандидатные коды в `taxonomy.dispute_type_code`, не форсировать классификацию через неподходящий существующий код и не создавать страницы по новой ситуации.

Пока не выделять в отдельные `dispute_type_code` без дополнительного основания:

- ст. 20–23 ЗоЗПП — сроки ремонта, замены и возврата денег по товару с недостатком; пока это подтипы `goods_defect_art18` + `claim_type_codes`/`situation_tags`;
- ст. 30–31 ЗоЗПП — сроки устранения недостатков/возврата денег по работам и услугам; пока это подтипы `work_service_defect_art29`, `work_service_delay_art28`, `service_refusal_art32`;
- ст. 13, 15, 17 ЗоЗПП — штраф, моральный вред, судебная защита; это требования/последствия, а не самостоятельные жизненные ситуации.

## Региональный и судебный слой

Регион и конкретный суд — самостоятельная ценность сервиса. Пользователь и юрист должны видеть, как похожие ситуации разрешаются в разных местах.

На старте:

- регионы и суды — метаданные, фильтры и аналитические признаки внутри страниц-ситуаций;
- отдельные страницы региона/суда не создаются автоматически;
- региональные и судебные различия выводятся в сводке только при достаточном количестве дел.

Ориентировочные пороги для будущих страниц:

| Тип страницы | Минимальный порог |
|---|---:|
| регион + категория ЗоПП | 15–20 дел |
| регион + конкретная ситуация | 7–10 дел |
| суд + категория | 10–15 дел |
| суд + конкретная ситуация | 5–7 дел, только если есть уникальная сводка |

До достижения порогов данные показываются внутри общей страницы ситуации.

## Инстанционный слой

Первичный глобальный фокус — суды общей юрисдикции.

Приоритет наполнения:

1. СОЮ первая инстанция — основная масса фактических жизненных историй.
2. Апелляция и кассация — подтверждение, корректировка или отмена подхода.
3. Верховный Суд РФ — ориентиры по важным вопросам.
4. Конституционный Суд РФ — если позиция имеет значение для ситуации.
5. Арбитражные суды и историческая практика ВАС РФ — второй глобальный этап, не смешивать с текущим гражданским ЗоПП-разделом без отдельной вертикали.

Для MVP не требуется идеальный баланс инстанций. Но для сильных ситуаций желательно постепенно добирать акты вышестоящих инстанций.

Ориентир по корпусу:

- 80–90% — первая инстанция;
- 10–20% — апелляция, кассация, ВС РФ, КС РФ по мере доступности.

## Цель первого публичного наполнения

Ориентир первой публичной версии:

- 150–250 валидных дел ЗоПП;
- 8–12 страниц-ситуаций;
- 3–5 сильных страниц с 20+ делами;
- 5–7 средних страниц с 6–15 делами;
- региональные и судебные срезы пока как фильтры/аналитика, не отдельные SEO-страницы;
- вышестоящие инстанции — точечно по сильным ситуациям.

Такой объём уже выглядит как полезный сервис, но не превращает проект в неконтролируемый массив.

## Практический порядок наполнения

1. Закрыть текущую незавершённую партию и довести базу до валидного состояния.
2. Пересчитать фактическое покрытие:
   - по `taxonomy.dispute_type_code`;
   - по регионам;
   - по судам;
   - по инстанциям;
   - по `publication.index_policy` / `publication.main_site_fit`.
3. Выделить слабые, но перспективные ситуации.
4. Следующие партии парсить по матрице:

```text
ситуация → недостающие регионы/суды → при возможности вышестоящие инстанции
```

5. Новые страницы создавать только при достижении порога полезности.

## Что не делать на старте

- Не гнаться за десятками тысяч страниц до проверки качества и индексации.
- Не создавать отдельные страницы региона/суда без плотности данных.
- Не смешивать СОЮ и арбитражные суды в одном пользовательском ЗоПП-разделе.
- Не расширять словарь ситуаций под единичные необычные дела.
- Не публиковать сомнительные дела без `hold`/ручной проверки.
