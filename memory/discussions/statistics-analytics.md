# Принять ли продуктовый инкремент «статистика/аналитика по делам»

## Постановка вопроса

Принять ли статистику/аналитику по судебным актам (суммы взыскания, винрейт
потребителя, срезы по регионам/судам/оппонентам, частоты норм) как продуктовое
направление; если да — какие метрики приоритетны для MVP, каковы пороги
выборки, как сочетать с E-E-A-T и SEO/индексацией, и что требует доработки
данных. Брейншторм-основа — `memory/topic-statistics-analytics.md`.

## Контекст проекта (уже сложилось)

- Большая часть нужных полей уже есть в JSON v2 (`claims_and_result.amounts`,
  `result`, `court.region/court`, `taxonomy`, `legal_analysis.legal_refs`).
- Расширение концепции на две аудитории (граждане + юристы) уже зафиксировано
  в `memory/topic-positioning.md`; профессиональная ценность там описана как
  «систематизация низовой судебной практики, обобщающие показатели».
- Анти-галлюцинация — критическое правило; «статистика» при малых выборках —
  родственный риск (не выдумки, но дезинформация).
- ТЗ (строки 88–94) уже задаёт принцип: служебные/срезные страницы
  индексируются только при достаточном числе дел и уникальном содержании.

### [ZCode] — 2026-06-26

**Позиция: принять, но поэтапно.** Направление сильное и технически зрелое:
данных в JSON v2 почти достаточно, решающий фактор не «можно ли», а «как
показать без ущерба E-E-A-T и SEO при малых n». Препонов на уровне данных почти
нет; риски сосредоточены в подаче и порогах.

**1. Приоритет метрик для MVP аналитики (от дешёвого к дорогому):**
- **Винрейт потребителя (% удовлетворения)** — самая дешёвая и самая
  «продающая» метрика; поле `result` уже структурировано.
- **Суммы по headline-категориям**: штраф 50% (п.6 ст.13 ЗоЗПП), неустойка
  (ст.23/23.1/28), моральный вред (ст.15) — медиана + диапазон. Не среднее
  (его ломают выбросы).
- **Соотношение «заявлено / присуждено»** — сильный инсайт, но требует обеих
  сумм в данных.
- В нормах/доводах и длительности рассмотрения — отложить до подтверждения
  гипотезы и доработки данных.

**2. Порог выборки (конкретное предложение — на критику в Раунде 2):**
- `n ≥ 30` в срезе → публикуем числа без оговорок.
- `15 ≤ n < 30` → показываем с явной плашкой «предварительно, малая выборка».
- `n < 15` → числа не показываем вовсе (только качественные формулировки).
Мотив: в YMYL-нише «статистика» на 5–10 делах — дезинформация, подрывающая
E-E-A-T, который проект строит целенаправленно.

**3. Две поверхности подачи:**
- **Для граждан** — лёгкие in-page метрики на страницах-ситуациях («в N%
  случаев потребителю возвращают деньги», «типичный штраф — X ₽»). Часть уже
  есть в прототипе как сводные чипы; это органичное расширение, не новый
  раздел.
- **Для юристов** — отдельные аналитические лонгриды/дашборды с granular-срезами
  по судам/регионам/оппонентам. Это и есть «профессиональная ценность» из
  `topic-positioning.md`.

**4. SEO/индексация — твёрдая линия:**
- **Не** генерировать массово срезы `region × category × claim` как отдельные
  индексируемые URL — прямой риск «Scaled Content Abuse» и пессимизации.
- Вместо этого: **немногочисленные сильные статические лонгриды** с уникальным
  авторским текстом + **интерактивный JS-дашборд** поверх предсчитанных JSON
  (без отдельных индексируемых URL). Это согласуется с ТЗ (строки 88–94) и
  снимает риск.

**5. Готовность данных — гейтить метрику, а не направление:**
- Суммы часто в свободном тексте → нужна нормализация валюты.
- «Заявлено/присуждено» требует обеих сумм.
- Длительность рассмотрения требует дат подачи (есть не всегда).
Эти пробелы должны блокировать конкретную метрику, но не всё направление.

**6. Порядок строительства:** начать с того, что текущие 50 актов уже
поддерживают (винрейт, суммы штрафов) → валидировать гипотезу «аналитика
привлекает профессиональную аудиторию» на одном сильном лонгриде → затем
инвестировать в нормализацию данных под остальные метрики.

**Главный риск:** стоимость разработки дашборда при непроверенном спросе со
стороны юристов. Митигация — начать со статического лонгрида (дёшево) и
проверить интерес, прежде чем строить интерактив.
