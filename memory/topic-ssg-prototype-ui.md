# SSG-прототип: UI и рендер

Этот topic-файл хранит актуальные правила и решения по интерфейсу статического
SSG-прототипа. Хронология и временные рабочие заметки здесь не хранятся — для
этого есть ежедневные файлы и `memory/current-work-queue.md`.

## Источник правок

- UI страниц меняется через единый генератор `scripts/generate_ssg_prototype.py`
  и пересборку `site_prototype/`.
- Ручные правки сгенерированных HTML-страниц не используются: они теряются при
  следующей генерации и ломают единообразие.
- Новые блоки нужно строить на существующих стилях (`panel`, `grid`,
  `summary-list`, `chips`, общие CSS-переменные). Новые CSS-правила добавлять
  только если существующего набора явно недостаточно.

## Страница дела

- Главная ценность страницы дела — понятная жизненная история, итог суда,
  практические выводы и подробный правовой разбор.
- Подзаголовки в блоке `История дела простым языком` должны выглядеть
  единообразно независимо от формата, который выдал агент: `### Кто участвовал`
  и строка `Кто участвовал:` рендерятся как один и тот же `<h4>Кто участвовал</h4>`.
  Нормализация выполняется в генераторе, а не ручной правкой `user_story_*.md`.
- Все `user_story_*.md` должны содержать стандартные подзаголовки. Это
  проверяется общим data-test:
  `validate_structures.py --strict-user-story-format --strict-md-consistency`.
- Блок `Подробный правовой разбор` должен быть единообразным для legacy/new
  `practice_*.md`: различия формата Markdown нормализуются на уровне рендера,
  без механического переписывания исходных `practice_*.md`.
- `practice_*.md` должен относиться к тому же `docid`, что и `structure_*.json`
  и `user_story_*.md`. Базовый автоматический контроль — `npm run test:data`
  (`validate_structures.py --strict-user-story-format --strict-md-consistency`): он ловит явные признаки
  подмены, например чужую организацию в правовом разборе при отсутствии якорей
  текущего дела.
- Старые строки норм вида `**Норма**: значение. Суд применил...` допустимо
  раскладывать на `Значение в деле` и `Применение судом` только если это можно
  сделать без добавления новых юридических выводов.
- Тот же legacy-рендер применяется к строкам норм вида `**Норма** — значение`:
  тире и двоеточие считаются допустимыми разделителями, а пояснение нормы
  выводится как `.md-note` с левой вертикальной линией.
- Вложенный legacy-формат норм вида `* **Норма**` + вложенные пункты
  `**Значение:** ...` / `**Применение:** ...` тоже нормализуется на уровне
  рендера: короткие метки приводятся к `Значение в деле` и `Применение судом`,
  а пояснения выводятся как `.md-note` с левой вертикальной линией.
- Генератор не должен добавлять слабую заглушку `Суд сослался на эту норму при разрешении спора`
  вместо отсутствующего `Применение судом`: это маскирует недоразметку.
  `data/review/practice-norm-application-debt.txt` должен оставаться пустым после переобработки
  известных случаев; новые случаи ловит `npm run test:data`.
- Если акт только перечисляет норму и не раскрывает отдельное применение, допустима
  нейтральная формулировка без домысливания: `Суд привёл эту норму в составе правового
  обоснования, но отдельно её применение не раскрыл.`
- Разделы `Нормы, на которые сослался суд`, `Логика решения`,
  `Факторы, повлиявшие на исход` приводятся к единой иерархии заголовков.
- В `practice_*.md` не смешивать markdown-заголовок и нумерацию: формат вида
  `### 1. Нормы, на которые сослался суд` запрещён. Для основных секций правового
  разбора использовать обычные заголовки без номера: `# Нормы, на которые сослался суд`,
  `# Логика решения`, `# Факторы, повлиявшие на исход`.
- В русских правовых аббревиатурах не использовать латиницу, похожую на кириллицу:
  писать `ГК`, `ГПК`, `АПК`, `КАС`, `ВС`, а не `GК`, `GПК`, `AПК`, `KАС`, `BС`.
  Это проверяется в `npm run test:data`.
- `claims_and_result.remedy` выводится в блоке `Что требовал потребитель`, поэтому
  это пользовательский русский текст, а не enum/служебное поле. Если значение выглядит
  как англоязычный служебный список (`refund`, `penalty`, `loan interest` и т.п.),
  генератор должен показать fallback из русских `taxonomy.claim_type_codes`, а
  `npm run test:data` должен падать для индексируемого JSON.
- Если на странице дела нет блока `Что произошло по датам`, но в блоке
  `Что требовал потребитель` есть оба денежных подблока (`Состав требований` и
  `Что присуждено`), эти подблоки на desktop выводятся в две внутренние колонки
  как отдельные вложенные карточки с рамкой. Это делает блок компактнее без
  визуального слияния колонок и без изменения общей сетки страницы; на мобильной
  ширине колонки складываются в одну.
- `Логика решения` рендерится как нумерованный список.
- `Факторы, повлиявшие на исход` рендерятся как обычный список, наследующий
  общий стиль страницы. Отдельные карточки, кастомные маркеры и лишние отступы
  не использовать без отдельного UI-решения.
- Если в legacy-тексте фактора нет явного короткого заголовка, генератор может
  осторожно выделить жирным уже существующее начало фразы до естественного
  разделителя. Новые слова или правовые выводы не добавляются.
- Блок, который выводит `case_summary.unusual_points` или
  `legal_analysis.arguments_rejected`, называется нейтрально:
  `Что важно учитывать`. Не использовать заголовок `Ошибки и риски`, потому что
  поле может содержать не только ошибки сторон, но и особенности акта, ограничения
  данных или нестандартные обстоятельства.

## Страница ситуации

- Страница ситуации должна давать пользователю не только список дел, но и
  сводку по текущей выборке: результаты, требования, объекты, компании/платформы,
  типичные факторы и рекомендации.
- Фильтры работают внутри страницы и не должны выглядеть как отдельные
  индексируемые URL.
- Карточки дел должны вести к странице дела заметной ссылкой/кнопкой.
- Заголовок карточки дела должен быть кратким и пользовательским, а не длинной
  цитатой фабулы.

## URL и маршруты

- ЗоПП-раздел реализован под `/zpp/`: `/zpp/`,
  `/zpp/praktika/<situation>/`, `/zpp/dela/<docid>/`.
- Корень `/` — общая входная страница сервиса «Судпрактика Онлайн».
- Прототипные маршруты `/praktika/...` и `/dela/...` больше не генерируются.
- При изменении маршрутов нужно одновременно обновлять генератор, внутренние
  ссылки, canonical URL, `sitemap.xml`, smoke/visual-тесты и проверки маршрутов.
- После публикации такие переносы делать только с редиректами, поэтому решение
  нужно выполнить до первого публичного релиза.

## Проверки после UI-изменений

- Пересобрать SSG через `scripts/generate_ssg_prototype.py`.
- CSS подключается с query-версией по хэшу содержимого `prototype.css`, чтобы
  браузер и локальный сервер не показывали устаревшие стили после пересборки.
- Проверить минимум одну страницу ситуации и одну страницу дела через локальный
  Docker-сервер.
- Прогнать `scripts/validate_structures.py`; при изменениях правового рендера
  также прогнать `scripts/verify_all.py`.
- Для визуально значимых правок после внедрения автотестов использовать
  smoke/visual regression tests из `memory/topic-visual-regression-testing.md`.
