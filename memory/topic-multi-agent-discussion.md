# Многостороннее обсуждение между ИИ-агентами

## Назначение
Воспроизводимый процесс коллективного разбора вопроса несколькими ИИ-агентами
(ZCode, Codex, MiMoCode, Antigravity) через общую файловую память проекта.
Агенты не имеют прямого канала связи — обмен идёт только файлами. Контракт
описывает процедуру; он кросс-проектен по природе и после обкатки переносится
в глобальную память (см. раздел «Промоушен контракта»).

## Роли
- **Инициатор** — пользователь: ставит вопрос и создаёт/открывает thread.
- **Участник** — агент, дающий ответ/рецензию в своём раунде.
- **Судья** — назначенный агент, синтезирует итог; не отвечает в раундах этого
  thread.
- **Оркестратор-gate** — пользователь: передаёт thread следующему агенту и
  принимает финальное решение (accept/reject/modify). Агент не передаёт
  управление автоматически — только предлагает следующий шаг.

## Топологии (поле `topology` в frontmatter thread'а)
- `star` — Раунд 1: каждый участник отвечает «вслепую» (видит только постановку).
  Подходит для быстрых мало-рисковых вопросов.
- `hybrid` (дефолт для нетривиальных) — Раунд 1 как в star + Раунд 2: каждый
  видит все ответы Раунда 1, проверяет факты, оспаривает, дополняет. Даёт
  кросс-проверку (второй слой анти-галлюцинации для юридического контента).
- `chain` — последовательный relay; использовать редко (anchoring bias, влияние
  порядка), явно помечать причину выбора во frontmatter.

## Процедура для агента, получившего thread
1. Прочитать frontmatter thread'а: `status`, `round`, `topology`, `role`,
   `resolves_to`, релевантный контекст по ссылкам.
2. Определить свой шаг по `round` + `topology`:
   - **Раунд 1 (мнение)**: ответить на вопрос самостоятельно, видя только
     постановку. Не читать чужие ответы Раунда 1.
   - **Раунд 2 (рецензия)**: прочитать все ответы Раунда 1, проверить факты,
     отметить ошибки/пробелы/галлюцинации, дополнить, при необходимости
     скорректировать.
   - **Вердикт (судья)**: синтезировать наиболее полное и эффективное решение
     из Раундов 1+2; не добавлять новых фактов без опоры на ответы/контекст.
3. Дописать свою секцию в thread под канонической подписью
   (`### [AgentName] — дата`), НЕ редактируя чужие секции. Один агент — одна
   секция за раунд.
4. Обновить `status`/`round` в frontmatter и **предложить пользователю следующий
   шаг** с инструкцией о дальнейших действиях (передать агенту X / перейти к
   Раунду 2 / передать судье). Управление не передаёт — ждёт решения пользователя.

## Анти-паттерны
- Не редактировать и не удалять чужие ответы.
- Не плодить дубликаты; если твоя секция за этот раунд уже есть — не добавлять вторую.
- Не изобретать факты/ссылки; в юридическом контенте — только из первоисточников
  и контекста проекта (действует анти-галлюцинационное правило из assistant-memory.md).
- Не передавать управление следующему агенту автоматически.

## Источник истины и жизненный цикл
Дискуссии (`memory/discussions/<slug>.md`) — **временные процессные артефакты**;
каноническое решение живёт в целевом topic-файле (указан в `resolves_to` во
frontmatter thread'а), который редактируется и эволюционирует по ходу проекта.

Жизненный цикл thread'а: `open` → `round-1` → (hybrid) `round-2` → `verdict` →
`resolved`. По достижении `resolved` синтез судьи **промоутируется в целевой
topic-файл** (решение + краткое обоснование — это часть стандартного контракта
topic-файлов проекта, без отдельной оговорки), после чего thread помечается
`resolved`. Сохранять ли дискуссию как аудит-лог или удалить — на усмотрение
инициатора; дефолт — сохранять (дёшево, мало, хранит rationale), удаление
допустимо после фиксации substance в topic-файле.

## Конверт шага (frontmatter thread'а)
`status`, `round`, `topology`, `participants`, `judge`, `resolves_to`,
`related`. Этого достаточно, чтобы любой агент, открыв thread, понял свой шаг
без отдельного status-файла на MVP. Автоматический status-файл
(`memory/discussions/_state.json`) потребуется только при переходе к
автоматической оркестрации (MVP процесса — ручная передача человеком).

## Отражение в очереди задач
Каждое открытое обсуждение отражается в `memory/current-work-queue.md` в разделе
«Активные обсуждения» строкой: ссылка на thread + текущая стадия (`round-1`/
`round-2`/`verdict`) + `resolves_to`. При смене стадии — обновить строку; при
`resolved` — убрать (решение промоутируется в topic-файл). Это делает подхват
обсуждения нативным для любого агента при восстановлении контекста, без
отдельного реестра.

## Самодостаточность thread'а
Поскольку контракт процедуры закреплён в памяти проекта, а каждое обсуждение
отражено в очереди задач со ссылкой на thread, инициатору **не нужно каждый раз
заново объяснять агенту суть вопроса**. Достаточно направить агента к thread'у
стандартной командой (см. шаблон в `README.md`, секция «Многостороннее
обсуждение»): агент сам читает frontmatter (`status`, `topology`, `resolves_to`,
`related`), постановку вопроса и контекст, после чего определяет свой шаг по
контракту. Подробная постановка и контекст живут в самом thread'е и его
`resolves_to`/`related` — там единственное место для их изложения; в очереди
хранится только указатель + стадия.

## Промоушен контракта в глобальную память
Контракт (процедура) кросс-проектен. После технологической обкатки на проекте
он переносится в глобальную память
(`C:\Users\Stas\.codex\memories\multi-agent-deliberation.md`, с указателем из
`memory-architecture.md`) для переиспользования на новых хостах и в новых
проектах. После переноса проект оставляет указатель на глобальный источник и
фиксирует только проектные отклонения — не держит вторую редактируемую копию во
избежание расхождения.

## Открытые вопросы
1. Перенос в глобальную память — после обкатки на 1–2 реальных обсуждениях.
2. Автоматический `_state.json` — только при переходе к автоматической
   оркестрации (вне MVP процесса).
