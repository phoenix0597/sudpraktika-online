# Дедупликация при парсинге судебных актов

## Решение

Для MVP используем лёгкий файловый реестр, без БД.

Канонические источники истины:

- `data/raw_acts/act_<docid>.txt`
- `data/raw_acts/act_<docid>.meta.json`
- `data/structured/structure_<docid>.json`

Генерируемый индекс:

- `data/registry/case_registry.csv`

Реестр строится скриптом `scripts/build_case_registry.py` из уже имеющихся raw/structured-артефактов. Он нужен для быстрого исключения уже известных дел при новых поисковых партиях и для передачи контекста внешним агентам.

## Ключи дедупликации

1. `docid` из URL вида `https://sudact.ru/regular/doc/<docid>/` — основной ключ до fetch.
2. Нормализованный `source_url` без хвостового `/` — резервный ключ до fetch.
3. `raw_text_sha256` — контрольный ключ после fetch, чтобы видеть фактическое совпадение текста.

## Текущий pipeline

1. Перед новым поиском обновить реестр:
   `python scripts/build_case_registry.py`
2. `scripts/find_cases.py` по умолчанию читает `data/registry/case_registry.csv` и исключает уже известные `docid`/URL из нового CSV.
3. Если нужен аудитный поиск без исключения известных дел, использовать `--include-known`.
4. `scripts/extract_act_text.py` в batch-режиме дополнительно пропускает уже существующие `data/raw_acts/act_<docid>.txt` и при необходимости дописывает `.meta.json`.
5. После новой загрузки raw-актов снова запустить `scripts/build_case_registry.py`.

`new_candidates.csv` должен содержать только фактически новые `docid` после дедупликации. Сырые результаты поиска хранятся в `search_*.csv`; если в `new_candidates.csv` попали уже известные `docid` или пересечения с другим активным batch-каталогом, это считается ошибкой отчёта партии, даже если повторный fetch потом был пропущен.

## Текущее состояние

На момент фиксации решения:

- в реестре 217 `docid`;
- все 217 имеют raw-текст;
- все 217 имеют `structure_<docid>.json`;
- batch `2026-06-27-zpp-next-2` показал практический риск: 32 из 50 строк пересекались с предыдущим batch-каталогом. Поэтому новые партии должны проверять пересечения не только с реестром, но и с активными/недавними batch-каталогами до фиксации `new_candidates.csv`.

## Масштабирование

CSV-реестр эффективен для ближайшего этапа и десятков тысяч строк, если он загружается один раз в память как `set(docid)`/`set(url)`. Проверка кандидата после этого почти константная.

Чтобы не деградировать на больших партиях, `scripts/find_cases.py` при наличии непустого `data/registry/case_registry.csv` читает только CSV. Сканирование `data/raw_acts/` и `data/structured/` используется только как fallback, если реестр ещё не создан.

## Почему пока не БД

Для текущего масштаба БД преждевременна: у нас уже есть стабильные файловые артефакты, а основной риск — повторный fetch и смешение партий — закрывается CSV-реестром. К БД имеет смысл возвращаться, когда появятся требования к конкурентной записи, сложным запросам, пользовательской админке или тысячам/десяткам тысяч актов.

Если парсинг станет действительно массовым или параллельным, следующий постоянный слой — SQLite:

- `docid` — unique index;
- `normalized_source_url` — unique index;
- `raw_text_sha256` — index;
- статусы: `found`, `fetched`, `queued`, `structured`, `hold`, `error`.

Redis не использовать как единственный источник истины. Он может быть полезен позже как runtime-слой для очередей, локов и атомарного резервирования `docid` несколькими воркерами, но постоянный реестр должен оставаться в файловом/SQLite-слое.
