# Судпрактика Онлайн — каталог судебной практики

Каталог судебной практики простым бытовым языком.
Первый этап проекта — дела о защите прав потребителей; архитектура рассчитана на дальнейшее расширение правовой тематики.
Генератор трафика → монетизация через РСЯ → масштабирование через новые разделы судебной практики.

Канонические требования — `technical-specification-mvp.md`.
Долговременные правила — `assistant-memory.md`.
Текущая очередь работы — `memory/current-work-queue.md`.

## Локальный просмотр сайта

Статический прототип можно смотреть через Docker Compose:

```bash
docker compose up -d
```

После запуска сайт доступен по адресу:

```text
http://localhost:8080/
```

Контур состоит из двух сервисов:

- `site` — отдаёт файлы из `site_prototype/`;
- `caddy` — входная точка, через которую потом можно включить домен и автоматический HTTPS.

Для production-настроек скопируйте `.env.example` в `.env` и укажите домен, например:

```text
SITE_ADDRESS=sudpraktika-online.ru
SITE_HTTP_PORT=80
SITE_HTTPS_PORT=443
SITE_PUBLIC_URL=https://sudpraktika-online.ru
```

Caddy сможет получить публичный SSL-сертификат, когда DNS домена указывает на сервер, а входящие порты 80 и 443 открыты.

`SITE_PUBLIC_URL` используется генератором статических страниц для canonical URL, `sitemap.xml` и `robots.txt`. Генератор читает это значение из переменной окружения или из `.env`. Если домен изменится, достаточно поменять значение и пересобрать сайт.

Остановить локальные контейнеры:

```bash
docker compose down
```

## Автотесты прототипа

Минимальный контур проверки UI работает через Playwright и локальный Docker-сервер.

```bash
npm install
npm run playwright:install
npm run test:all
```

Отдельные команды:

```bash
npm run test:data           # structured-артефакты, формат user_story и защита от рассинхрона markdown с делом
npm run test:citations      # проверка правовых ссылок на галлюцинации по всем practice-файлам
npm run test:ci             # полный CI-контур: data + citations + пересборка SSG + UI-тесты
npm run test:smoke          # маршруты, обязательные блоки, ссылки, sitemap/robots
npm run test:visual         # сравнение с эталонными скриншотами
npm run site:test           # полный UI-прогон smoke + visual
npm run test:all            # data-test + полный UI-прогон
npm run test:visual:update  # обновить эталонные скриншоты после намеренного изменения UI
```

Visual snapshots обновляйте только после проверки, что изменения интерфейса ожидаемые.

GitHub Actions использует тот же CI-контур через `.github/workflows/ci.yml` и сохраняет Playwright-отчёт как artifact.

## Как извлечь структуру из судебного акта (инструкция для ИИ-агента)

Когда нужно обработать собранные судебные акты — запускаете ИИ-агента
(**Antigravity** с Gemini-3.5-flash, либо ZCode) в корневой папке проекта.
Агент читает готовую инструкцию и действует по ней. Вам не нужно формулировать
задачу с нуля — достаточно направить агента к файлу инструкции.

### Где лежит инструкция агента

```
data/inbox/_TASK.md      ← ЕДИНАЯ ИНСТРУКЦИЯ для ИИ-агента
```

В этом файле описано всё: роль, три задачи (пользовательская история,
правовой анализ и JSON-структура дела), критические правила
(анти-галлюцинация, стиль без приветствий), формат входа/выхода,
самопроверка.

### Что сказать агенту (шаблон команды)

Откройте Antigravity (или ZCode) в папке проекта и дайте команду:

> Прочитай и выполни инструкцию в `data/inbox/_TASK.md`.

Этого достаточно. В инструкции описано всё: роль, три задачи, какие акты брать
(pending из `_queue.json`), куда класть результат, как обновить статус.
Агент сам:
1. прочтёт `_TASK.md` (роль, задачи, правила),
2. возьмёт pending-акты из `_queue.json`,
3. прочитает тексты актов из `data/raw_acts/act_<docid>.txt`,
4. запишет три файла в `data/structured/`:
   - `user_story_<docid>.md` — бытовая история,
   - `practice_<docid>.md` — правовой анализ,
   - `structure_<docid>.json` — структурированные данные дела,
5. проставит `status: done` в `_queue.json` только после полного комплекта.

### После работы агента — проверка на галлюцинации

Вывод любого агента проверяется единым верификатором (не зависит от модели):

```bash
py scripts/verify_citations.py --act data/raw_acts/act_<docid>.txt --analysis data/structured/practice_<docid>.md
```

Верификатор сверяет каждую правовую ссылку с текстом акта и флагует отсутствующие
(подозрение на галлюцинацию). Цель: 0 выдуманных ссылок.

Полнота JSON и комплекта файлов проверяется отдельно:

```bash
py scripts/validate_structures.py
```

### Как добавить новые акты в очередь

После сбора новых актов (через `scripts/find_cases.py` + `scripts/extract_act_text.py`)
зарегистрируйте их в очереди:

```bash
py scripts/queue_for_agent.py            # регистрирует все акты из data/raw_acts/
py scripts/queue_for_agent.py --status   # показать статус очереди
```

## Многостороннее обсуждение (для ИИ-агентов)

Когда нужно коллективно разобрать вопрос несколькими ИИ-агентами (ZCode, Codex,
MiMoCode, Antigravity) через общую файловую память проекта — используется
воспроизводимый контракт обсуждения. Агенты обмениваются мнениями файлами,
каждый подписывает свой ответ, человек передаёт пайплайн на каждом шаге, финал —
резюме агента-судьи.

### Где лежит контракт

```
memory/topic-multi-agent-discussion.md      ← КОНТРАКТ обсуждения (процедура)
memory/discussions/<slug>.md                ← конкретные обсуждения (временные)
```

Контракт описывает роли (инициатор, участник, судья, человек-gate), топологии
(`star` / `hybrid` / `chain`), процедуру шага агента, анти-паттерны, правило
«каждый агент даёт пользователю инструкции при передаче пайплайна» и жизненный
цикл. Каноническое решение обсуждения промоутируется в целевой topic-файл
(указан в `resolves_to` во frontmatter конкретной дискуссии).

### Как запустить

Откройте нужного агента в корне проекта и дайте команду обычным языком:

> Восстанови контекст проекта, найди активное обсуждение по теме
> «<тема обсуждения>» и подключись к нему по действующему контракту.

Этого достаточно: агент сам найдёт обсуждение в очереди/памяти проекта,
определит текущий раунд, запишет подписанный ответ и вернёт пайплайн вам с
инструкцией.

Если конкретный thread-файл уже известен, его можно указать напрямую, но это
необязательно.

## Структура данных

```
data/
├── raw_acts/       ← сырые тексты актов: act_<docid>.txt
├── structured/     ← извлечённые структуры (агент): user_story_<docid>.md, practice_<docid>.md, structure_<docid>.json
└── inbox/          ← зона обмена с ИИ-агентом
    ├── _TASK.md    ← инструкция для агента (читать первым)
    └── _queue.json ← очередь актов (pending → done)

scripts/            ← код пайплайна (Python)
output/             ← временные выводы скриптов (не в репозитории)
memory/             ← рабочая память проекта (заметки, решения)
```

Файловый ключ акта = **doc-id источника** (напр. `yQ8qgPoesvWJ` из sudact.ru URL),
а НЕ номер дела (номер не уникален между регионами).

## Скрипты пайплайна

| Скрипт | Назначение |
|---|---|
| `scripts/find_cases.py` | Поиск судебных дел через Yandex Search API (оператор `site:sudact.ru`) |
| `scripts/extract_act_text.py` | Извлечение чистого текста акта из HTML sudact.ru |
| `scripts/extract_structure.py` | Извлечение JSON-структуры через DeepSeek API |
| `scripts/backfill_case_metadata.py` | Восстановление source-метаданных и заготовок JSON v2 для уже скачанных актов |
| `scripts/complete_structures_deepseek.py` | Дозаполнение незавершённых `structure_<docid>.json` через DeepSeek API |
| `scripts/generate_enum_suggestions.py` | Генерация предложений по нормализации enum для ручной проверки |
| `scripts/run_prompt.py` | Прогон произвольного промпта по акту (DeepSeek) |
| `scripts/queue_for_agent.py` | Наполнение очереди обработки для ИИ-агента |
| `scripts/validate_structures.py` | Проверка полноты комплекта user_story/practice/structure и валидности JSON |
| `scripts/verify_citations.py` | Проверка ссылок на галлюцинации (анти-галлюцинация) |
| `scripts/wordstat_fetch.py` | Сбор данных спроса через Yandex Wordstat API |
| `scripts/aggregate_demand.py` | Агрегация спроса по вертикалям |
| `scripts/hybrid_lookup.py` | Гибридный поиск фактов через DeepSeek API (экономия MCP-квоты) |

Промпты LLM — в `scripts/`: `prompt_extract.txt`, `prompt_user_story.txt`,
`prompt_practice_analysis.txt`.

Словари нормализованных значений для `dispute_type_code`, `claim_type_codes`,
`result_type` и `publication.index_policy` — в `data/reference/zpp_enum_dictionary.json`.

## Ключевые правила проекта

1. **Анти-галлюцинация:** правовые ссылки — только из текста самого акта.
   Запрещено генерировать ссылки на внешние дела/постановления из «знаний» модели.
2. **Стиль ответа LLM:** без приветствий и мета-фраз, только содержание.
3. **MCP-квота:** веб-поиск/чтение — только в крайнем случае; обычно DeepSeek API + HTTP.
4. **Память:** сырые данные в `data/`, заметки/решения в `memory/`, не смешивать.

Подробно — в `assistant-memory.md` и `technical-specification-mvp.md`.
