## Архитектурные Решения

- Проектная рабочая память должна использовать легкую трехслойную схему: ежедневные заметки в `memory/YYYY-MM-DD.md`, повторяющиеся тематические заметки в `memory/topic-<slug>.md` и сжатая долговременная память проекта в `assistant-memory.md`.
- Ежедневные файлы должны оставаться короткими и ссылаться на тематические файлы, когда обсуждения становятся повторяющимися, архитектурно значимыми или слишком длинными для краткой хронологической заметки.
- Схема проектной памяти должна оставаться совместимой с общей межпроектной архитектурой из `.codex/memories/memory-architecture.md` относительно домашнего каталога текущего пользователя: глобальные правила в глобальном каталоге памяти, долговременные проектные правила в `assistant-memory.md`, рабочий проектный контекст в `memory/`.

## Продукт и Стратегия

- **Глобальная цель проекта:** масштабируемый каталог судебной практики ЗоПП, генерирующий стабильный органический трафик и конвертирующий его в доход, в первую очередь через РСЯ — без публичного лица и без оказания юридических услуг. Подробности — в `technical-specification-mvp.md`. Числовые вехи трафика — в `memory/topic-product-strategy.md`.
- Продукт: каталог судебной практики по защите прав потребителей простым бытовым языком на основе массового парсинга судебных актов. НЕ ИИ-сервис консультаций (исходная гипотеза отозвана при планировании).
- Канонические требования к продукту, модели данных, схеме извлечения и структуре страниц — в `technical-specification-mvp.md`.
- Обоснование решений, риски, дорожная карта (с Фазой 0 — валидационным гейтом) и исследовательская база — в `memory/topic-product-strategy.md`.
- Источник данных — официальный доступ к отдельным судебным актам со страниц судов в ГАС «Правосудие» (sudrf.ru); программное обращение к конкретным актам по реквизитам/ссылкам, НЕ извлечение базы целиком. Единичный акт не охраняется АП (ст. 1225 ГК РФ); право изготовителя базы (ст. 1334 ГК РФ) защищает компиляцию, а не единичный документ — поэтому не затрагивается. Суды публикуют акты уже обезличенными в силу Закона №262-ФЗ — повторное обезличивание не требуется. Правовая оценка подтверждена пользователем-юристом 2026-06-24.
- **Обнаружение дел (Шаг 2, доказано 2026-06-24):** Yandex Search API (`searchapi.api.cloud.yandex.net/v2/web/search`) на ключе Wordstat-API; тело `{query:{search_type:"SEARCH_TYPE_RU", query_text:...}}`. Возвращает XML в base64 с URL'ами реальных дел (напр. sudact.ru). Подходит для поиска релевантных актов по теме.
- **Навык «гибридный веб-поиск» (закреплён 2026-06-24):** для фактов/знаний/enum/конфигов использовать DeepSeek API (`scripts/hybrid_lookup.py`) → прицельный HTTP (Bash) → MCP веб-поиск/чтение только в крайнем случае (max 1–2/сессию).
- **Лимиты Yandex Search API (проверено 2026-06-25):** для web/search синхронная квота — 10 000 запросов/час и 10 запросов/сек; максимум выдачи — 250 результатов; запрос — до 400 символов / 40 слов. Для Wordstat отдельно: 100 запросов/час и 10 запросов/сек. Для партий 50–100 актов лимиты web/search достаточны; в скриптах всё равно нужен delay/retry на 429/5xx.
- **Структура данных проекта (рабочее соглашение, закреплено 2026-06-24):** постоянные данные живут в `data/` (в репозитории, переживают очистку): `data/raw_acts/` — сырые тексты актов, `data/structured/` — извлечённые структуры. Временные выводы скриптов — в `output/` (.gitignore). Рабочая память (`memory/`) — только заметки/решения, НЕ склад сырых данных. Промпты LLM — в `scripts/` рядом с кодом.
- **Ключ дела (закреплено 2026-06-24):** файловый ключ = **doc-id источника** (ID документа из URL sudact.ru), а НЕ номер дела. Номер дела не уникален (одноимённые дела в разных судах/регионах → конфликт файлов). Пример: `act_yQ8qgPoesvWJ.txt` (doc-id из sudact.ru/regular/doc/yQ8qgPoesvWJ/). Номер дела, суд, регион, дата — в метаданных JSON для поиска/дедупликации. Если акты будут из других источников (sudrf.ru) — doc-id этого источника.
- **Анти-галлюцинация LLM (критическое правило, закреплено 2026-06-25):** LLM извлекает правовые ссылки (статьи/дела/постановления/пункты) ТОЛЬКО из текста акта. Запрещено генерировать ссылки на внешнюю практику/дела/пункты из «знаний» модели — это главная угроза для юридического сайта (выдуманные цитаты = дезинформация, разрушение доверия). Двойная защита: (1) промпт с явным запретом (`scripts/prompt_practice_analysis.txt`), (2) программный верификатор `scripts/verify_citations.py` сверяет каждую ссылку с текстом акта, флагует отсутствующие. Проверено на деле 2-80/2025: v1 промпт = 4 галлюцинации (выдуманное СПб дело, ФЗ-250, ФЗ-234); v2 с запретом = 0 галлюцинаций.
- **Стиль ответа LLM (закреплено 2026-06-25):** промпты запрещают приветствия и мета-фразы («Здравствуйте, я проанализировал...», описания процесса) — недопустимы в финальном контенте страниц. Отвечать только содержанием.
- **Словари enum для JSON v2 (согласовано 2026-06-25):** канонический словарь нормализованных значений — `data/reference/zpp_enum_dictionary.json`. `taxonomy.dispute_type_code` строится преимущественно от правовой конструкции ЗоЗПП (`goods_defect_art18`, `prepaid_goods_delay_art23_1`, `distance_sale_return_art26_1`, `info_violation_art10_12`, `work_service_defect_art29`, `work_service_delay_art28`, `service_refusal_art32` и т.п.), а не от маркетплейса/товара. Ситуационные детали хранятся в `platform_or_company`, `object_type`, `object_name`, `situation_tags`. `taxonomy.claim_type_codes` — список способов защиты/требований (возврат цены, замена, ремонт/устранение недостатков, уменьшение цены, неустойка, убытки, моральный вред, штраф 50%, расходы и т.п.).
- **Извлечение через Antigravity/DeepSeek (гибридный путь, закреплено 2026-06-25; уточнено по JSON 2026-06-25):** внешний ИИ-агент Antigravity (Gemini-3.5-flash) используется как дешёвый масштабируемый путь, но при исчерпании его лимитов незавершённые `structure_<docid>.json` добиваются через DeepSeek API на актуальной модели `deepseek-v4-pro` (`scripts/complete_structures_deepseek.py`). Интеграция через «почтовый ящик»: Python наполняет `data/inbox/_queue.json` (`scripts/queue_for_agent.py`) и детерминированно фиксирует source-метаданные (`source_url`, домен, путь к акту, SHA-256 текста) при парсинге/бекфилле. Агент/скрипт пишет полный комплект в `data/structured/`: `user_story_<docid>.md` + `practice_<docid>.md` + `structure_<docid>.json`. Статус `done` допустим только после полного комплекта; JSON считается финальным только при `processing.status: "complete"`. Вывод гоняется через `verify_citations.py`/`verify_all.py` и `validate_structures.py`, включая проверку правовых ссылок внутри JSON.
- E-E-A-T закрывается через профиль безликой «юридической редакции» + методологию + верифицируемость по номерам дел. Публичное лицо проекта отсутствует по выбору пользователя.
- **Многостороннее обсуждение между ИИ-агентами (закреплено 2026-06-26):** воспроизводимый процесс коллективного разбора вопроса агентами (ZCode, Codex, MiMoCode, Antigravity) через общую файловую память. Процедура (роли, топологии `star`/`hybrid`/`chain`, шаг агента, человек-gate на каждом переходе, жизненный цикл, промоушен в topic-файл) описана в `memory/topic-multi-agent-discussion.md`; конкретные обсуждения живут во временных `memory/discussions/<slug>.md`. После обкатки контракт переносится в глобальную память для переиспользования. Это правило не дублируется здесь полностью — см. topic-файл.
- Монетизация (старт): **РСЯ/Я.Директ как основной и фокусный канал** (решение 2026-06-24). Стратегия: генератор трафика → РСЯ → масштабирование расширением правовой тематики. Лиды юристам + партнёрки + DOCX-генератор — отложены/пост-MVP (не активируются на старте).
- **Пилотный сегмент (решение 2026-06-24 по данным Wordstat):** маркетплейсы (Озон/ВБ) + процессуальные ситуации (неустойка/претензия/иск) — спрос 112K + 107K показов/мес. Ранее пилотом были технически сложные товары (спрос 16K) — опровергнуто замером. Сырые данные — `memory/topic-demand-raw-data.md`, анализ — `memory/topic-market-sizing.md`.
- Масштаб запуска поэтапный: пилот 100–300 страниц, затем +500–1000/мес. Темп ограничен сознательно против риска Scaled Content Abuse.

## Рабочие Соглашения Проекта

Общие правила обновления контекста после шага, единого канонического источника и предпочтения ссылок — в `.codex/memories/global-collaboration-rules.md` относительно домашнего каталога; они действуют и здесь. Ниже только то, что специфично для проекта.

- При возобновлении по короткой команде пользователя вроде "продолжим" сначала восстанови контекст по стартовым файлам, проверь текущий этап и согласованность статусных файлов. Если новых фактов или задач нет, не создавай новый артефакт; достаточно подтвердить текущий этап и исправить устаревшие статусные строки при необходимости.
- Список улучшений памяти в `memory/topic-memory-improvement.md` держи видимым и управляемым: фиксируй наблюдение, предлагаемое правило, статус и место закрепления; не считай кандидат правилом до закрепления в `assistant-memory.md`, `AGENTS.md` или другом подходящем источнике правды.
- В repo-local memory-файлах используй относительные пути от корня репозитория. Для глобальной памяти используй путь от домашнего каталога текущего пользователя и раскрывай его в формат активной оболочки только при выполнении команд.
- Сообщения коммитов (commit messages) в репозитории проекта всегда пишутся на русском языке.
